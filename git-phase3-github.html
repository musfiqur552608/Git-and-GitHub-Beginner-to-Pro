Lesson 01 of 06 · Phase 3 — GitHub & Remotes

GitHub setup & SSH keys

Before you can push code to GitHub, you need two things: an account and a way to authenticate. SSH keys are the professional, password-free way to connect. Set this up once and never think about it again.

~30 minone-time setupcritical
github account essentials
step 01
Create account
Go to github.com. Use the same email you set in git config user.email. Choose a professional username — it'll be your developer identity.
step 02
Set up your profile
Add a photo, bio, and location. Enable two-factor authentication (2FA) — GitHub requires it for pushing code.
step 03
SSH or HTTPS?
SSH: key-based, no password prompts, recommended. HTTPS: simpler to start, requires a Personal Access Token (PAT) instead of password.
step 04
Verify connection
Run ssh -T git@github.com after setup. You should see: "Hi username! You've successfully authenticated."
generate & add an SSH key
terminal — SSH key setup (do this once)
# Step 1: Generate a new ED25519 key (modern, faster than RSA) $ ssh-keygen -t ed25519 -C "you@example.com" Generating public/private ed25519 key pair. Enter file in which to save the key (~/.ssh/id_ed25519): [press Enter] Enter passphrase (recommended): •••••••• Your identification has been saved in ~/.ssh/id_ed25519 Your public key has been saved in ~/.ssh/id_ed25519.pub # Step 2: Start the SSH agent $ eval "$(ssh-agent -s)" Agent pid 12345 # Step 3: Add your key to the agent $ ssh-add ~/.ssh/id_ed25519 # Step 4: Copy your PUBLIC key to clipboard $ cat ~/.ssh/id_ed25519.pub # macOS/Linux: print it $ clip < ~/.ssh/id_ed25519.pub # Windows: copy to clipboard $ pbcopy < ~/.ssh/id_ed25519.pub # macOS: copy to clipboard # Step 5: Go to GitHub → Settings → SSH and GPG keys → New SSH key # Paste your PUBLIC key there. Title it (e.g. "MacBook 2025") # Step 6: Test the connection $ ssh -T git@github.com Hi yourusername! You've successfully authenticated.
💡
Public vs Private key: Your id_ed25519.pub (public) goes on GitHub — share freely. Your id_ed25519 (private, no .pub) stays on your machine ONLY. Never share or expose the private key.
⚠
Windows users: Use Git Bash (not PowerShell/CMD) for SSH setup. Or enable the OpenSSH agent in Windows Services. The clip command works in Git Bash to copy keys.
personal access tokens (PAT) — HTTPS alternative

If you use HTTPS instead of SSH, GitHub no longer accepts your account password for Git operations. You need a Personal Access Token (PAT) — use it wherever a password is asked.

creating a PAT
# Go to: GitHub → Settings → Developer settings → Personal access tokens # → Tokens (classic) → Generate new token # Select scopes: repo, workflow, read:org # Copy the token immediately — you won't see it again! # Store it with Git credential helper so you don't retype it: $ git config --global credential.helper store # Next time you push, enter your username + PAT as password. # Git will remember it after the first successful push.
exercise — connect your machine to github
Generate SSH key and connect to GitHub setup
  • 1Create a GitHub account at github.com (if you don't have one)
  • 2Enable 2FA: Settings → Password and authentication → Two-factor authentication
  • 3Run the SSH key generation commands above step by step
  • 4Add the public key to GitHub: Settings → SSH and GPG keys → New SSH key
  • 5Run ssh -T git@github.com — you should see your username in the output
  • 6Go to GitHub, create a new empty repo called my-journal. Don't initialize with README.
  • 7Follow the "push an existing repo" commands GitHub shows you after creation

Lesson 02 of 06 · Phase 3 — GitHub & Remotes

push · pull · fetch

These three commands connect your local Git repo to the world. Understanding the difference between fetch and pull is one of the most important Git nuances — and most developers get it wrong for years.

~40 mincore commandsdaily use
the remote model — how git tracks remotes

When you clone a repo, Git creates a remote tracking branch — a local read-only copy of the remote branch at the time of your last communication. This is what origin/main means: "where main was on origin last time we talked."

understanding remotes
# List all remotes (URL + name) $ git remote -v origin git@github.com:you/my-journal.git (fetch) origin git@github.com:you/my-journal.git (push) # Add a remote to an existing local repo $ git remote add origin git@github.com:you/my-journal.git # See local branches vs remote tracking branches $ git branch -vv * main 4f2a1b3 [origin/main] update day 3 notes feature abc1234 [origin/feature: ahead 2] add weather # "ahead 2" = you have 2 local commits not yet pushed # "behind 3" = remote has 3 commits you haven't pulled yet
git push — sending local commits to github
git push — all variations
# First push — set upstream tracking with -u $ git push -u origin main Branch 'main' set up to track remote branch 'main' from 'origin'. To git@github.com:you/my-journal.git * [new branch] main → main # After -u is set, just use git push $ git push # Push a feature branch (for Pull Requests) $ git push -u origin feature/login # Delete a remote branch $ git push origin --delete feature/login # Push all branches at once $ git push --all origin # Force push (DANGEROUS — only on your own branches) $ git push --force-with-lease # safer than --force
🚫
Never force push to main/shared branches. --force rewrites remote history and breaks everyone who has pulled. If you must force push your own feature branch, use --force-with-lease — it checks nobody else pushed since your last pull.
fetch vs pull — the critical difference
git fetch — safe, look before you leap
  • Downloads remote commits to origin/main
  • Does NOT touch your local branch at all
  • You inspect changes, then merge manually
  • Use when: you want to see what changed remotely without integrating yet
  • Zero risk of conflicts or broken code
git pull — fetch + merge in one step
  • Downloads AND integrates remote changes
  • Equivalent to: fetch + merge (or + rebase)
  • Your working directory is updated immediately
  • Use when: you trust the remote and want to sync fast
  • Can cause conflicts if you have uncommitted local changes
fetch vs pull in practice
# ── FETCH (safe) ────────────────────────────────────── $ git fetch origin # download all remote updates $ git log origin/main --oneline # see what they did $ git diff main origin/main # diff your local vs remote $ git merge origin/main # merge when you're ready # ── PULL (convenient) ───────────────────────────────── $ git pull # fetch + merge (default) $ git pull --rebase # fetch + rebase (cleaner history) $ git pull origin main # explicit: from origin, branch main # ── FETCH WITH PRUNE ────────────────────────────────── $ git fetch --prune # remove deleted remote branch refs
💡
Pro habit: Use git pull --rebase by default. It avoids unnecessary merge commits when syncing with the remote, keeping history linear. Set it globally: git config --global pull.rebase true
exercise — push your journal to github
Push my-journal to GitHub and practice sync easy
  • 1Go to github.com → New repository → name it my-journal → Create (no README)
  • 2In your local repo: git remote add origin git@github.com:YOURUSERNAME/my-journal.git
  • 3Push: git push -u origin main — your commits appear on GitHub!
  • 4On GitHub, edit README.md directly in the browser. Commit the change there.
  • 5Locally: git fetch origin — then git log origin/main --oneline to see the remote commit
  • 6Merge it: git merge origin/main — now your local is in sync
  • 7Bonus: create a feature branch, push it, and find it on GitHub under "branches"
Troubleshooting: If push is rejected with "failed to push some refs", someone else (or you via GitHub editor) pushed commits you don't have. Run git pull --rebase first, then push again.
Lesson 03 of 06 · Phase 3 — GitHub & Remotes

Fork & Pull Request workflow

The fork-and-PR model is how all open source collaboration works — and how most professional teams collaborate on GitHub. Master this and you can contribute to any project on Earth.

~50 minteam workflowindustry standard
fork vs clone — what's the difference?
Fork (GitHub server-side copy)
  • Creates YOUR copy of someone else's repo on GitHub
  • You have full write access to your fork
  • Used to contribute to repos you don't own
  • Stays connected to the original (upstream)
  • Click "Fork" button on GitHub — that's it
Clone (local copy)
  • Downloads any repo (yours or forked) to your machine
  • You need write permission to push back
  • Always after forking: git clone <your-fork-url>
  • Sets origin to your fork
  • You manually add upstream remote for the original

complete fork workflow — every step
# 1. Fork on GitHub (click the Fork button) — creates: # github.com/YOU/original-repo # 2. Clone YOUR fork (not the original) $ git clone git@github.com:YOU/original-repo.git $ cd original-repo # 3. Add the original repo as "upstream" $ git remote add upstream git@github.com:OWNER/original-repo.git $ git remote -v origin git@github.com:YOU/original-repo.git (fetch/push) upstream git@github.com:OWNER/original-repo.git (fetch/push) # 4. Create a feature branch for your changes $ git switch -c fix/typo-in-readme # 5. Make your changes, commit $ git add README.md $ git commit -m "docs: fix typo in installation section" # 6. Push your branch to YOUR fork (not upstream) $ git push -u origin fix/typo-in-readme # 7. Open a Pull Request on GitHub # GitHub shows a "Compare & pull request" banner — click it! # OR use the GitHub CLI: $ gh pr create --fill # ── STAYING IN SYNC WITH UPSTREAM ────────────────────── # While waiting for your PR to be reviewed: $ git fetch upstream $ git rebase upstream/main # or: git merge upstream/main $ git push --force-with-lease # update your PR branch
writing a good pull request
PR template — what to include
## What does this PR do? Fixes the login button not responding on mobile Safari (iOS 16+). The issue was a CSS `pointer-events: none` applied too broadly. ## Why? Closes #142 — bug reported by 3 users this week. ## How to test 1. Open app on iOS Safari 2. Tap the Login button 3. Should navigate to /dashboard (was broken before this fix) ## Screenshots Before | After [broken screenshot] [working screenshot] ## Checklist - [x] Tests pass - [x] No console errors - [x] Tested on iOS Safari 16 and Chrome Mobile
💡
Small PRs merge faster. Keep PRs focused on one thing. A 50-line PR gets reviewed in 5 minutes. A 1000-line PR sits for days. One feature or fix per PR is the golden rule.
exercise — simulate the full fork-to-PR cycle

  • 2Click Fork. Clone your fork: git clone git@github.com:YOU/first-contributions.git
  • 3Add yourself to Contributors.md on a feature branch: git switch -c add-your-name
  • 4Commit and push to your fork: git push -u origin add-your-name
  • 5Open the PR on GitHub. Fill in a description. Submit it.
  • 6Alternatively: create a second GitHub account, fork YOUR own my-journal, and submit a PR to yourself. Review and merge it.
Pro move: After your PR is merged, sync your fork with upstream: git fetch upstream && git merge upstream/main && git push

Code review on GitHub

Code review is where most professional collaboration actually happens. Learn to give reviews that help, receive feedback without ego, and use GitHub's review tools like a pro.

~35 minteam skill
github review types — what each means
PR review states explained
✓ Approved ← "Looks good, merge when ready" — LGTM ✗ Request changes ← "Fix these things before merging" ○ Comment ← "I have notes but not blocking the merge" Types of review comments: Nitpick (nit): "nit: prefer const over let here" Suggestion: Actual code suggestion the author can apply with one click Question: "Why did you choose this approach over X?" Blocker: "This will cause a race condition, must fix before merge"
giving good code review — the rules
✓ good review habits
  • → Review the code, not the person
  • → Be specific — "line 42: consider X because Y"
  • → Distinguish nitpicks from blockers
  • → Explain your reasoning, not just what
  • → Approve promptly — don't leave PRs hanging
  • → Use GitHub Suggestions for small fixes
✗ avoid these
  • → "This is bad" with no explanation
  • → Blocking on personal style preference
  • → Reviewing tone: "obviously this is wrong"
  • → Demanding rewrites for minor issues
  • → Leaving a PR open for days with no response
  • → Approving without actually reading
github review commands
reviewing & responding to PRs
# View PR diff locally $ gh pr checkout 42 # switch to PR branch locally $ gh pr diff 42 # view the diff in terminal $ gh pr view 42 # view PR details in terminal # Submit review via CLI $ gh pr review 42 --approve $ gh pr review 42 --request-changes -b "Please fix the auth bug on line 47" $ gh pr review 42 --comment -b "Looks mostly good, left one nit" # Merge a PR after approval $ gh pr merge 42 --squash # squash all commits into one $ gh pr merge 42 --merge # preserve commits with merge commit $ gh pr merge 42 --rebase # rebase onto base branch # After merging, clean up $ git switch main $ git pull $ git branch -d feature/my-branch
exercise — give and receive a code review
Simulate review: open a PR to yourself and review it medium

  • 2Open a Pull Request from feature/add-goals → main on GitHub
  • 3In the PR, click "Files changed" — practice adding a line comment on a specific line
  • 4Try "Add a suggestion" — edit the proposed code in the suggestion box. Submit your review.
  • 5Apply the suggestion by clicking "Commit suggestion" — watch the branch auto-update
  • 6Approve and merge the PR. Delete the branch via the GitHub UI.
  • 7Locally: git pull && git branch -d feature/add-goals
Lesson 05 of 06 · Phase 3 — GitHub & Remotes

Issues · Projects · Labels

GitHub is more than hosting — it's a project management platform. Issues track bugs and features. Projects organize them on a kanban board. Labels and milestones keep large projects sane.

~25 minproject management
issues — the backbone of project tracking

Every bug report, feature request, or task lives as a GitHub Issue. Issues are numbered (#1, #2…) and can be referenced in commits and PRs to automatically close them on merge.

linking commits & PRs to issues
# Closes issue #42 when this PR merges (magic keywords) $ git commit -m "fix: resolve login null crash — closes #42" # Other keywords that auto-close issues: # fix, fixes, fixed, close, closes, closed # resolve, resolves, resolved # Reference an issue without closing it $ git commit -m "refactor: improve auth module (related to #38)" # Create an issue via CLI $ gh issue create --title "Login broken on Safari iOS" --label bug $ gh issue list # list open issues $ gh issue view 42 # view issue details $ gh issue close 42 # close an issue
standard github labels
🐛 bug ✨ enhancement 📖 documentation 🆘 help wanted 🌱 good first issue ⛔ wontfix ❓ question 🔁 duplicate
💡
good first issue is the label to look for when you want to contribute to open source. Maintainers use it to mark beginner-friendly issues. Filter by it on any large repo to find your first contribution.
github projects — kanban boards

GitHub Projects (v2) gives you a spreadsheet-like board for tracking issues and PRs. You can create custom fields, group by status, and automate card movement.

example kanban board

exercise — set up issues and labels
Create issues, link them in commits, and auto-close easy
  • 1In your my-journal GitHub repo, go to Issues → New Issue. Create one called "Add weekly summary section"
  • 2Label it "enhancement". Note its issue number (e.g. #3)
  • 3Create a branch: git switch -c feature/weekly-summary
  • 4Add a weekly.md file. Commit with: git commit -m "add weekly summary — closes #3"
  • 5Push and open a PR. Merge it. Go back to Issues — #3 should be automatically closed!
  • 6Bonus: go to Projects tab, create a new Project, and add 3 issues to it. Drag them through the board columns.
