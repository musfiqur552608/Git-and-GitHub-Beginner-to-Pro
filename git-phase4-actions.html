Lesson 01 of 06 · Phase 4 — GitHub Actions & CI/CD

GitHub Actions & CI/CD

GitHub Actions automates everything that happens around your code — tests run automatically on every push, code is deployed when PRs merge, and releases are created on version tags. No manual steps, no "it works on my machine."

~25 minconcept2025 must-know
what is CI/CD?
CI — Continuous Integration
Every code push automatically triggers: build the project, run all tests, check code quality. If anything breaks, you know in seconds — not days.
CD — Continuous Delivery/Deployment
After CI passes, code is automatically deployed to staging or production. When main merges → users get the update. No manual SSH, no FTP uploads.
💡
Why it matters: Teams using CI/CD ship 200x more frequently with 24x lower failure rates (2023 State of DevOps report). It's the biggest single skill that elevates a developer's career right now.
github actions core concepts
ConceptWhat it is
WorkflowA YAML file in .github/workflows/ — defines the automation. One repo can have many workflows.
Event (trigger)What starts the workflow: push, pull_request, schedule, release, workflow_dispatch (manual)
JobA group of steps that run on the same runner. Jobs run in parallel by default.
StepA single command or Action. Steps within a job run sequentially.
RunnerA virtual machine (ubuntu-latest, windows-latest, macos-latest) that runs your jobs
ActionA reusable unit from the Marketplace (e.g. actions/checkout, actions/setup-node)
SecretEncrypted environment variable stored on GitHub — for API keys, passwords, tokens
ArtifactFiles produced by a workflow (build output, test reports) that can be downloaded
your first workflow — hello github actions
.github/workflows/hello.yml
name: Hello GitHub Actions on: push: branches: [main] jobs: greet: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Say hello run: echo "Hello from GitHub Actions! 🎉" - name: Show the current date run: date - name: List repo files run: ls -la
✓
Try it right now: Create .github/workflows/hello.yml in any repo, paste this, commit and push. Go to the Actions tab on GitHub within 10 seconds — you'll see it run live.

Lesson 02 of 06 · Phase 4 — GitHub Actions & CI/CD

YAML syntax & triggers

Every Actions workflow is a YAML file. YAML is whitespace-sensitive — indentation matters. Master the structure once and every workflow becomes readable.

~40 mincore syntax
yaml crash course — the 5 rules
yaml rules every developer needs to know
# Rule 1: Key-value pairs with colon-space name: My Workflow # string value timeout-minutes: 30 # number value continue-on-error: false # boolean value # Rule 2: Indentation = 2 spaces (never tabs) jobs: my-job: # 2 spaces runs-on: ubuntu-latest # 4 spaces steps: # 4 spaces - run: echo hi # 6 spaces (list item) # Rule 3: Lists use dash-space branches: - main - develop # OR inline: branches: [main, develop] # Rule 4: Multiline strings run: | # pipe = literal block (keeps newlines) echo "line 1" echo "line 2" npm test # Rule 5: Expression syntax ${{ }} for dynamic values run: echo "Branch is ${{ github.ref_name }}"
all the important triggers (on:)
workflow triggers — all common patterns
# ── PUSH TRIGGER ───────────────────────────────────────── on: push: branches: [main, develop] # only these branches paths: # only when these files change - 'src/**' - 'package.json' # ── PULL REQUEST TRIGGER ───────────────────────────────── on: pull_request: branches: [main] # PRs targeting main types: [opened, synchronize, reopened] # ── SCHEDULE (CRON) ────────────────────────────────────── on: schedule: - cron: '0 9 * * 1-5' # 9am UTC, Mon–Fri - cron: '0 0 * * 0' # midnight Sunday # ── MANUAL TRIGGER ─────────────────────────────────────── on: workflow_dispatch: # "Run workflow" button on GitHub inputs: environment: description: 'Deploy to?' default: 'staging' required: true # ── RELEASE TRIGGER ────────────────────────────────────── on: release: types: [created] # when a GitHub Release is published # ── MULTIPLE TRIGGERS ──────────────────────────────────── on: [push, pull_request, workflow_dispatch]
github context — dynamic values inside workflows
useful context variables
# github context — info about the triggering event ${{ github.sha }} # full commit SHA (e.g. a1b2c3d...) ${{ github.ref_name }} # branch name (main, feature/x) ${{ github.actor }} # username who triggered the workflow ${{ github.repository }} # owner/repo-name ${{ github.event_name }} # push, pull_request, etc. ${{ github.run_number }} # incrementing run count (1, 2, 3...) # env context — environment variables ${{ env.MY_VAR }} # access any env: variable you defined # secrets context — encrypted secrets ${{ secrets.MY_SECRET }} # never printed in logs (masked) # Example usage steps: - name: Tag Docker image run: docker build -t app:${{ github.sha }} .
exercise — create a workflow with multiple triggers
Write a workflow triggered by push AND manual dispatch easy
  • 1In your my-journal repo, create folder: mkdir -p .github/workflows
  • 2Create .github/workflows/info.yml with triggers: push (main only) and workflow_dispatch
  • 3Add a job with steps that print: branch name, actor, commit SHA, and current date
  • 4Use ${{ github.ref_name }}, ${{ github.actor }}, ${{ github.sha }} in your echo commands
  • 5Commit and push. Go to Actions tab — watch it run automatically
  • 6Click "Run workflow" button (from the workflow_dispatch trigger) and trigger it manually too
YAML gotcha: Indentation errors are the #1 cause of workflow failures. Use 2 spaces everywhere. Never mix tabs and spaces. VS Code's YAML extension (redhat.vscode-yaml) catches these before you push.
Lesson 03 of 06 · Phase 4 — GitHub Actions & CI/CD

Jobs, steps & runners

This is where workflows get powerful. Multiple jobs run in parallel. Jobs can depend on each other. And you can run the same job across 9 OS/version combinations simultaneously with a matrix.

~45 mincore architecture
parallel jobs with dependencies
.github/workflows/pipeline.yml — job dependencies
name: CI Pipeline on: [push, pull_request] jobs: # Job 1: runs first (no needs:) lint: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm run lint # Job 2: also runs first (parallel with lint) test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' - run: npm ci && npm test # Job 3: waits for BOTH lint AND test to pass build: runs-on: ubuntu-latest needs: [lint, test] # ← dependency declaration steps: - uses: actions/checkout@v4 - run: npm run build # Job 4: only runs on main branch, after build deploy: runs-on: ubuntu-latest needs: build if: github.ref == 'refs/heads/main' # ← conditional steps: - run: echo "Deploying to production!"
matrix strategy — test across multiple versions

A matrix runs the same job across multiple configurations simultaneously. Instead of writing 9 jobs for Node 18/20/22 × ubuntu/windows/macos, write one job with a matrix.

matrix strategy — 9 jobs from one definition
jobs: test: strategy: matrix: os: [ubuntu-latest, windows-latest, macos-latest] node: [18, 20, 22] fail-fast: false # don't cancel others if one fails runs-on: ${{ matrix.os }} # dynamic runner from matrix name: Node ${{ matrix.node }} on ${{ matrix.os }} steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node }} - run: npm ci - run: npm test # Result: 3 OS × 3 Node versions = 9 parallel jobs # All run simultaneously — total time ≈ time of slowest single job
caching dependencies — speed up your pipeline
cache node_modules between runs
steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' # ← built-in caching! - run: npm ci # ~30s first run, ~3s cached runs - run: npm test # Manual cache control - uses: actions/cache@v4 with: path: ~/.npm key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }} restore-keys: | ${{ runner.os }}-node-
💡
Cache key strategy: Hash your lockfile (package-lock.json, requirements.txt) as the cache key. When dependencies change, the hash changes, cache misses, and fresh packages install. Perfect invalidation automatically.
exercise — build a multi-job pipeline
Create a pipeline with lint → test → build jobs medium
  • 1Create a Node.js project: npm init -y && npm install eslint jest --save-dev
  • 2Add a sum.js function and a sum.test.js test. Add npm test to package.json scripts.
  • 3Create .github/workflows/ci.yml with 3 jobs: lint, test, build
  • 4Make build depend on both lint and test using needs: [lint, test]
  • 5Push. Go to Actions → watch the DAG (directed acyclic graph) of jobs run in parallel then converge
  • 6Bonus: intentionally break a test, push, and watch the pipeline go red. Fix it and watch it go green.
Lesson 04 of 06 · Phase 4 — GitHub Actions & CI/CD

Secrets & environments

API keys, deploy tokens, and passwords should never be in your code. GitHub Secrets encrypts them at rest and injects them at runtime — they're never visible in logs.

~30 minsecurity
interactive secrets manager
Repository secrets — Settings → Secrets → Actions
VERCEL_TOKEN••••••••••••••••••••Updated 3 days ago
DATABASE_URL••••••••••••••••••••Updated 1 week ago
SLACK_WEBHOOK••••••••••••••••••••Updated 2 weeks ago
🚫
NEVER echo a secret in a workflow run command. GitHub automatically masks secrets in logs — but if you base64-encode or transform them, the mask may not work. Treat secrets as write-only.
using secrets in workflows
secrets usage patterns
# Pass secret as environment variable (most common) steps: - name: Deploy to Vercel env: VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }} run: vercel deploy --prod # Pass to an Action's input - uses: actions/deploy@v2 with: token: ${{ secrets.GITHUB_TOKEN }} # built-in token api-key: ${{ secrets.MY_API_KEY }} # Repo-level vs Environment-level secrets # Repository: available to all workflows # Environment: only available when deploying to that env jobs: deploy-prod: environment: production # gates on required reviewers steps: - env: PROD_KEY: ${{ secrets.PROD_API_KEY }} # env-level secret run: ./deploy.sh production
github environments — staging & production gates
environment protection rules
# Settings → Environments → production → Protection rules: # ✓ Required reviewers: @yourteam-leads # ✓ Wait timer: 5 minutes # ✓ Deployment branches: main only jobs: deploy-staging: environment: staging # deploys immediately runs-on: ubuntu-latest steps: - run: echo "Deploying to staging..." deploy-production: needs: deploy-staging environment: production # waits for manual approval! runs-on: ubuntu-latest steps: - run: echo "Deploying to production!"
✓
Production gates are gold. Setting required reviewers on your production environment means no code reaches production without a human approving it — even if the automated pipeline would have deployed it.
Lesson 05 of 06 · Phase 4 — GitHub Actions & CI/CD

Real CI pipeline

A production-grade CI workflow that you can use directly in any project. Tests, linting, coverage, security scan, and artifacts — all automated.

~50 minproduction-grade
complete production ci workflow
.github/workflows/ci.yml — production-ready
name: CI on: push: branches: [main] pull_request: branches: [main] env: NODE_VERSION: '20' jobs: ### ── JOB 1: Lint & Format ───────────────────────────── lint: name: Lint & Format Check runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ env.NODE_VERSION }} cache: npm - run: npm ci - run: npm run lint - run: npm run format:check ### ── JOB 2: Tests with Coverage ────────────────────── test: name: Tests & Coverage runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ env.NODE_VERSION }} cache: npm - run: npm ci - run: npm test -- --coverage - name: Upload coverage report uses: actions/upload-artifact@v4 with: name: coverage-report path: coverage/ ### ── JOB 3: Security Audit ──────────────────────────── security: name: Security Audit runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm audit --audit-level=high ### ── JOB 4: Build (needs lint + test) ───────────────── build: name: Production Build needs: [lint, test, security] runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ env.NODE_VERSION }} cache: npm - run: npm ci && npm run build - name: Upload build artifact uses: actions/upload-artifact@v4 with: name: dist-${{ github.sha }} path: dist/ ### ── JOB 5: Deploy (main only) ──────────────────────── deploy: name: Deploy to Vercel needs: build runs-on: ubuntu-latest if: github.ref == 'refs/heads/main' && github.event_name == 'push' environment: production steps: - uses: actions/checkout@v4 - uses: amondnet/vercel-action@v25 with: vercel-token: ${{ secrets.VERCEL_TOKEN }} vercel-org-id: ${{ secrets.VERCEL_ORG_ID }} vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }} vercel-args: '--prod'
exercise — production ci for your own project
Build a full CI pipeline for a real Node.js or Python project hard
  • 1Create a small web project (React app: npx create-react-app, or Next.js: npx create-next-app)
  • 2Push it to GitHub. Set up the full CI workflow above (copy it, adjust paths)
  • 3Watch all 5 jobs run. Check the Artifacts section — download the coverage report.
  • 4Add a branch protection rule: Settings → Branches → main → Require status checks: test, lint must pass before merging
  • 5Create a PR that intentionally breaks a test. Watch GitHub block the merge with "Required checks failed"
  • 6Fix the test. Push. Watch CI go green. Now you can merge.
Branch protection rules are the real power move. Once set up, NO ONE can merge broken code to main — not even repo owners. This is how professional teams maintain code quality at scale.

Deploy to cloud

Connect your CI pipeline to the world. Deploy to Vercel, Netlify, Docker Hub, and AWS — all triggered automatically on merge to main.

~40 mindeployment
deploy to vercel (front-end projects)
vercel auto-deploy workflow
name: Deploy to Vercel on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Deploy to Vercel Production uses: amondnet/vercel-action@v25 with: vercel-token: ${{ secrets.VERCEL_TOKEN }} vercel-org-id: ${{ secrets.VERCEL_ORG_ID }} vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }} vercel-args: '--prod' # Get tokens from: vercel.com → Settings → Tokens # Get org/project IDs from: .vercel/project.json after first deploy
docker build & push to container registry
docker build + push workflow
name: Docker Build & Push on: push: branches: [main] tags: ['v*'] # also on version tags (v1.0.0) env: REGISTRY: ghcr.io IMAGE_NAME: ${{ github.repository }} jobs: build-push: runs-on: ubuntu-latest permissions: contents: read packages: write # needed for GHCR steps: - uses: actions/checkout@v4 - name: Log into GitHub Container Registry uses: docker/login-action@v3 with: registry: ${{ env.REGISTRY }} username: ${{ github.actor }} password: ${{ secrets.GITHUB_TOKEN }} - name: Build & push Docker image uses: docker/build-push-action@v5 with: context: . push: true tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} cache-from: type=gha # GitHub Actions cache cache-to: type=gha,mode=max

Phase 4 quick reference

All GitHub Actions concepts, triggers, and Marketplace actions in one place.

workflow structure
KeyDescription
name:Workflow name (shown in GitHub UI)
on:Event triggers — push, pull_request, schedule, workflow_dispatch, release
env:Workflow-level environment variables (accessible to all jobs)
jobs:Top-level map of jobs. Jobs run in parallel unless needs: is specified
runs-on:Runner OS: ubuntu-latest, windows-latest, macos-latest
needs:Job dependencies — list of jobs that must pass first
if:Conditional expression — job only runs if true
steps:Ordered list of commands and actions within a job
uses:Reference a reusable action from Marketplace
run:Shell command to execute
with:Input parameters for an action
environment:Named environment with protection rules and secrets
