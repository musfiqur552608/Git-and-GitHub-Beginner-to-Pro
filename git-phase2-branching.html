Lesson 01 of 06 · Phase 2 — Branching & Merging

What is a branch?

A branch is Git's killer feature. It lets you work on new ideas in complete isolation — without touching the stable code. When you're ready, you bring the work back. The whole internet runs on this workflow.

~20 min concept most important in phase 2
the mental model

Think of your project history as a timeline of commits. A branch is just a movable pointer to one of those commits. When you commit on a branch, the pointer advances forward automatically.

HEAD is a special pointer that says "this is where you are right now." It points to your current branch, which in turn points to the latest commit on that branch.

branch as a lightweight pointer
c1 ──── c2 ──── c3 ← main (branch pointer) ↑ HEAD (you are here) After creating feature/login branch: c1 ──── c2 ──── c3 ← main ↑ feature/login ← HEAD After making 2 commits on the feature branch: c1 ──── c2 ──── c3 ← main \ c4 ──── c5 ← feature/login ← HEAD
💡
Key insight: A branch is just a 41-byte file containing a commit hash. Creating a branch costs almost nothing — no copying, no duplication. That's why branches are cheap and you should use them constantly.

Lesson 02 of 06 · Phase 2 — Branching & Merging

branch · switch · checkout

These are the commands you'll use to create, navigate, and manage branches. Git 2.23+ introduced switch and restore to replace the overloaded checkout — use the modern syntax.

~35 min hands-on daily commands
creating & switching branches
all the ways to work with branches
# ── LISTING BRANCHES ──────────────────────────────────── $ git branch # local branches (* = current) $ git branch -a # local + remote branches $ git branch -v # branches + last commit message # ── CREATING BRANCHES ──────────────────────────────────── $ git branch feature/login # create (but stay on current branch) $ git switch -c feature/login # create + switch in one step ✓ preferred $ git checkout -b feature/login # older syntax — same result # ── SWITCHING BRANCHES ─────────────────────────────────── $ git switch main # switch to main $ git switch - # switch to PREVIOUS branch (very handy) $ git checkout main # older syntax # ── DELETING BRANCHES ──────────────────────────────────── $ git branch -d feature/login # safe delete (only if merged) $ git branch -D feature/login # force delete (even if unmerged) # ── RENAMING A BRANCH ──────────────────────────────────── $ git branch -m old-name new-name
naming conventions — industry standard
PatternUse caseExample
feature/<desc>New feature workfeature/user-login
fix/<desc>Bug fixesfix/null-pointer-auth
hotfix/<desc>Urgent production fixhotfix/payment-crash
chore/<desc>Maintenance, refactoringchore/update-deps
docs/<desc>Documentation onlydocs/readme-update
release/<version>Release preparationrelease/v2.1.0
💡
Use lowercase-with-hyphens: No spaces, no uppercase. The slash creates visual grouping in tools like GitHub and GitKraken. Keep it short but descriptive.
git switch vs git checkout — what changed in 2.23
git switch (new — 2019+)
  • Only switches branches / creates branches
  • Clearer, unambiguous purpose
  • git switch -c <branch> to create
  • Can't accidentally checkout files
  • Recommended for all new users
git checkout (legacy)
  • Does branches AND file restoration
  • Confusing dual purpose
  • git checkout -b <branch> to create
  • Still works — won't be removed
  • You'll see it everywhere in old docs/Stack Overflow
✓
Recommendation: Use git switch going forward. It's clearer and less error-prone. When you see git checkout in tutorials, mentally translate it to git switch.
exercise — create your first feature branch
Create and work on a feature branch easy
  • 1In your my-journal repo: git switch -c feature/weather-log
  • 2Create a file: echo "# Weather Log" > weather.md then commit it
  • 3Run git log --oneline --all — notice your branch is ahead of main
  • 4Switch back to main: git switch main — notice weather.md disappears from your folder!
  • 5Switch back to the feature branch: git switch - — it comes back. This is the magic.
  • 6Run git branch -v to see all branches and their latest commit
Why does the file disappear? Git updates your working directory to match the branch you switch to. Your files are safe — they're stored as commits on the feature branch.
Lesson 03 of 06 · Phase 2 — Branching & Merging

Merging — the fast-forward

Merging brings the work from one branch into another. The simplest kind — a fast-forward merge — happens when there's a straight line of commits. No conflicts, no new merge commit.

~30 min core concept
fast-forward merge — when it happens

A fast-forward merge occurs when the target branch (usually main) has not moved since you branched off. Git can simply move the pointer forward — no new merge commit needed.

fast-forward merge — before & after
─── BEFORE MERGE ──────────────────────────────────── c1 ──── c2 ──── c3 ← main \ c4 ──── c5 ← feature/login ← HEAD ─── AFTER: git switch main && git merge feature/login ── c1 ──── c2 ──── c3 ──── c4 ──── c5 ← main ← HEAD ↑ feature/login # Output you see: Updating c3..c5 Fast-forward login.html | 47 +++++++++++++++++++ 1 file changed, 47 insertions(+)
merge with --no-ff — preserving history

In teams, it's common to use --no-ff (no fast-forward) to always create a merge commit. This preserves the fact that a feature branch existed — your history shows the branch structure clearly.

--no-ff merge — creates merge commit
# Switch to destination branch first $ git switch main # Regular fast-forward merge (default) $ git merge feature/login Fast-forward # Merge with explicit merge commit (team standard) $ git merge --no-ff feature/login -m "merge: add user login feature" Merge made by the 'ort' strategy. login.html | 47 ++++++ auth.js | 23 ++++ # Result with --no-ff: c1 ─── c2 ─── c3 ───────────────── M6 ← main \ / c4 ─── c5 ──/ ← feature/login (merged)
💡
GitHub does this automatically when you merge a Pull Request — it always creates a merge commit (or optionally squashes). This is why PRs are preserved in your history.
the complete branch + merge workflow
feature branch full lifecycle
# 1. Start from a clean main branch $ git switch main $ git pull # make sure you're up to date # 2. Create your feature branch $ git switch -c feature/login # 3. Do your work (any number of commits) $ git add login.html auth.js $ git commit -m "add login form HTML" $ git commit -m "add JWT auth logic" # 4. Switch back and merge $ git switch main $ git merge --no-ff feature/login -m "merge: user login feature" # 5. Clean up (delete the branch — it's already merged) $ git branch -d feature/login Deleted branch feature/login (was abc1234).

Lesson 04 of 06 · Phase 2 — Branching & Merging

3-way merge & conflict resolution

Merge conflicts are not failures — they're Git asking for your judgment when two branches changed the same part of a file. Every developer resolves them daily. Once you know the pattern, they're easy.

~45 min critical skill
when does a 3-way merge happen?

A 3-way merge happens when both branches have diverged — each has commits the other doesn't. Git finds a common ancestor commit and merges the three versions: ancestor, yours, theirs.

diverged history → 3-way merge
Both branches diverged (both have commits the other lacks): c1 ──── c2 ──── c3 ← main (added homepage.html) \ c4 ──── c5 ← feature/login (added auth.js) 3 reference points for the merge algorithm: [1] Common ancestor: c2 [2] "Ours" (main): c3 [3] "Theirs" (feature): c5 Result — a new merge commit: c1 ── c2 ── c3 ──────── M6 ← main \ / c4 ── c5 ─/
merge conflict — what it looks like

When Git can't automatically merge a file (both branches changed the same lines), it inserts conflict markers into the file and pauses, waiting for you to decide.

app.js — conflict markers explained
function greet(user) { <<<<<<< HEAD ← start of YOUR version (current branch) return `Hello, ${user.name}`; ← your code ======= ← separator return `Hi there, ${user.name}! 👋`; ← incoming branch code >>>>>>> feature/friendlier-ui ← end of THEIR version (incoming) }
⚠
Don't panic! Git hasn't broken your code. It's just showing you both versions and asking: "Which one do you want, or do you want something different?" You edit the file to remove the markers and keep what you want.
resolving conflicts — step by step
conflict resolution workflow
# 1. A merge conflict occurs $ git merge feature/friendlier-ui CONFLICT (content): Merge conflict in app.js Automatic merge failed; fix conflicts and then commit. # 2. Check which files conflict $ git status both modified: app.js # 3. Open the file(s) and edit manually # OR use VS Code's merge editor: # VS Code will highlight each conflict with Accept Incoming / # Accept Current / Accept Both buttons. # 4. After editing — remove ALL conflict markers <<< === >>> # Then stage the resolved file $ git add app.js # 5. Commit the merge (Git pre-fills the message) $ git commit # ── ABORT if you made it worse ────────────────────────── $ git merge --abort # go back to before the merge started

Lesson 05 of 06 · Phase 2 — Branching & Merging

git rebase — a cleaner history

Rebase re-applies your commits on top of another branch, giving you a clean linear history. It's one of Git's most powerful — and most misunderstood — features.

~40 min intermediate
merge vs rebase — the visual difference
merge vs rebase — same result, different history
─── STARTING POINT (both have same starting point) ─────── c1 ── c2 ── c3 ← main \ c4 ── c5 ← feature/login ← HEAD ─── AFTER git merge main ────────────────────────────────── c1 ── c2 ── c3 ───────── M6 ← main (merge commit) \ / c4 ── c5 ── ↑ preserves diverge history ─── AFTER git rebase main ───────────────────────────────── c1 ── c2 ── c3 ── c4' ── c5' ← feature/login ↑ c4 & c5 are replayed on top of c3 they get NEW commit hashes (c4', c5') but same content changes
git merge
  • Creates a merge commit (M6)
  • Preserves exact history of what happened
  • Safe to use on shared/public branches
  • History can look messy with many features
  • Best for: main branch, shared branches
git rebase
  • No merge commit — linear history
  • Rewrites commit hashes (creates new commits)
  • Never rebase shared/public branches
  • Clean, easy-to-read history
  • Best for: local feature branches before PR
🚫
The golden rule of rebase: Never rebase commits that exist outside your local repo (i.e., pushed to a shared branch). Rebase rewrites history — if others have already pulled those commits, you'll create chaos for the whole team.
interactive rebase — rewriting your own history

git rebase -i (interactive) is one of the most powerful things in Git. It lets you squash commits together, reorder them, edit messages, or delete commits entirely — all before sharing your work.

interactive rebase — clean up before PR
# You have 4 messy commits you want to clean up $ git log --oneline a7b1234 fix typo 9c3f891 oops forgot a file 4d2e567 WIP changes 1a8b234 add login feature # Open interactive rebase for last 4 commits $ git rebase -i HEAD~4 # Editor opens — change 'pick' to an action word: pick 1a8b234 add login feature squash 4d2e567 WIP changes ← squash into previous squash 9c3f891 oops forgot a file ← squash into previous reword a7b1234 fix typo ← just edit the message # After save, Git asks for the combined commit message # Result: 4 messy commits → 1 clean commit Successfully rebased and updated refs/heads/feature/login.
CommandWhat it does in interactive rebase
pick (p)Keep this commit as-is
reword (r)Keep commit, but edit its message
squash (s)Merge into previous commit, combine messages
fixup (f)Merge into previous commit, discard this message
drop (d)Delete this commit entirely
edit (e)Pause here to amend the commit
exercise — rebase a feature branch
Rebase a feature branch onto updated main medium
  • 1On main, make a new commit (simulate teammate activity): add a line to README, commit it
  • 2Create feature branch: git switch -c feature/rebase-test
  • 3Make 3 commits on this branch with messages like "wip 1", "wip 2", "done"
  • 4Now rebase onto main: git rebase main — your 3 commits replay on top
  • 5Run git log --oneline --graph — see the clean linear history
  • 6Now try interactive squash: git rebase -i HEAD~3. Change 2nd and 3rd to squash. Save — write a clean combined message.
Tip: If rebase gets confusing, git rebase --abort always gets you back to where you started. No damage done.

git stash — save work temporarily

You're halfway through a feature when your manager says "urgent bug fix needed on main NOW." Stash saves your unfinished work to a stack so you can switch contexts immediately — and come back later.

~25 min daily use
stash commands
git stash — all you need to know
# Save uncommitted work to the stash (with a description) $ git stash push -m "WIP: half-done login form" Saved working directory and index state On feature/login: WIP: half-done login form # List all stashed items $ git stash list stash@{0}: On feature/login: WIP: half-done login form stash@{1}: On main: quick experiment # Apply latest stash (and keep it in the stash list) $ git stash apply # Apply latest stash AND remove it from list $ git stash pop # Apply a specific stash $ git stash apply stash@{1} # Delete a specific stash $ git stash drop stash@{0} # Also stash untracked files $ git stash push -u -m "include new files too" # Create a new branch from a stash (very useful) $ git stash branch feature/recovered-work stash@{0}
