Lesson 01 of 06 · Phase 1 — Foundation

What is Git & why does it matter?

Before touching a single command, understand the mental model. Git is not a backup tool — it's a time machine for your code that also lets multiple people work on the same files without overwriting each other.

~20 min no tools needed concept
the problem git solves

Imagine you're writing an essay. You save copies as essay_v1.docx, essay_v2.docx, essay_final.docx, essay_final_REAL.docx… sound familiar? Git solves this by tracking every change you make with a message describing what changed and why.

💡
The core idea Git stores snapshots of your project over time, not just file differences. Each snapshot (commit) has a unique ID, author, date, and a message you write. You can jump to any snapshot at any time.
git vs github — they are NOT the same
Git
A free, open-source tool you install on your computer. It tracks changes in any folder on your machine. Works 100% offline.
GitHub
A website that hosts your Git repositories online. It adds collaboration features like pull requests, issues, and Actions on top of Git.
⚠
Common confusion: You can use Git without GitHub. GitHub requires Git. Think of Git as the engine and GitHub as the garage where you park it.
key vocabulary
TermWhat it means
repository (repo)A folder tracked by Git — contains all your files + their full history
commitA saved snapshot with a message, timestamp, and unique ID (hash)
branchA parallel version of your project — you'll learn this in Phase 2
working directoryThe folder on your disk where you actually edit files
staging areaA "prep zone" where you choose exactly which changes to include next commit
HEADA pointer to the commit you're currently on — usually the latest commit

Lesson 02 of 06 · Phase 1 — Foundation

Installing Git & first-time config

Get Git running on your machine and tell it who you are. These three config commands are the first thing every developer runs — you only do them once per computer.

~25 min hands-on setup
installing git
Windows
Download git-scm.com installer. Use default settings. Comes with Git Bash.
macOS
Run xcode-select --install or install via Homebrew: brew install git
Linux
Ubuntu/Debian: sudo apt install git. Fedora: sudo dnf install git
first-time global config

Git needs to know who you are before it can record commits. These settings are saved to ~/.gitconfig on your machine.

terminal
# Tell Git your name (appears in every commit you make) $ git config --global user.name "Your Full Name" # Tell Git your email (use the same one as GitHub later) $ git config --global user.email "you@example.com" # Set VS Code as your default Git editor $ git config --global core.editor "code --wait" # Set main as the default branch name (modern standard) $ git config --global init.defaultBranch main # Set color output (makes reading much easier) $ git config --global color.ui auto
verify your config
terminal
# List all your current config settings $ git config --list user.name=Your Full Name user.email=you@example.com core.editor=code --wait init.defaultbranch=main color.ui=auto # Check a specific value $ git config user.name Your Full Name # Check Git version (should be 2.40+) $ git --version git version 2.49.0
✓
Three config levels: --global applies to your user (all repos). --local applies to just the current repo. --system applies to all users on the machine. Global is what you want 99% of the time.
exercise — set up git on your machine
Configure Git for the first time easy
  • 1Open your terminal (Terminal on Mac/Linux, Git Bash on Windows)
  • 2Run git --version to confirm Git is installed. You should see a version number.
  • 3Run the 5 config commands above with your real name and email.
  • 4Run git config --list and confirm your name and email appear.
  • 5Click the checkbox above to mark this exercise complete.
Hint: On Windows, if git is not found, close and reopen your terminal after installation. The PATH needs to refresh.
Lesson 03 of 06 · Phase 1 — Foundation

git init, git clone & the working directory

Every Git repo starts one of two ways: you create a fresh one with git init, or you download an existing one with git clone. These are the two entry points to every project.

~30 min hands-on core command
what lives inside a git repo

When you run git init, Git creates a hidden .git/ folder. This is the entire history of your project — all commits, branches, config. Never manually edit anything inside .git/.

project structure after git init
my-project/ ├── .git/ ← Git's brain — don't touch this │ ├── HEAD ← points to current branch │ ├── config ← repo-specific config │ ├── objects/ ← all your commits & file data │ └── refs/ ← branches & tags ├── index.html ├── style.css └── README.md
git init — start a new repo
terminal
# Create a new folder and make it a git repo $ mkdir my-project $ cd my-project $ git init Initialized empty Git repository in /Users/you/my-project/.git/ # Or: create + init in one step $ git init my-project Initialized empty Git repository in my-project/.git/ # Confirm the hidden .git folder was created $ ls -la drwxr-xr-x .git
git clone — copy an existing repo
terminal
# Clone via HTTPS (simpler, asks for password) $ git clone https://github.com/user/repo.git # Clone via SSH (recommended once SSH is set up) $ git clone git@github.com:user/repo.git # Clone into a specific folder name $ git clone https://github.com/user/repo.git my-folder-name # Clone only the latest commit (faster for large repos) $ git clone --depth 1 https://github.com/user/repo.git Cloning into 'repo'... remote: Counting objects: 423, done. Receiving objects: 100% (423/423), 1.24 MiB | 4.2 MiB/s, done.
💡
init vs clone: Use git init for brand-new projects you're starting yourself. Use git clone when a repo already exists somewhere (GitHub, a colleague's machine, etc.).
exercise — create your first repo
Init a "my-journal" repo and explore its structure easy
  • 1Open your terminal and run: git init my-journal
  • 2Navigate into it: cd my-journal
  • 3Run ls -la (Mac/Linux) or dir /a (Windows) — you should see the .git folder.
  • 4Create a file: echo "# My Journal" > README.md
  • 5Run git status — you should see README.md listed as "untracked". This is expected! We'll commit it in Lesson 5.
Clone any public GitHub repo easy
  • 1Go to any public GitHub repo (e.g. github.com/torvalds/linux for fun, or something smaller)
  • 2Click the green "Code" button and copy the HTTPS URL
  • 3Run: git clone <URL> --depth 1 (depth 1 = only latest commit, much faster)
  • 4Run git log --oneline inside it — you'll see real commits from other developers

The 3 states — Git's core mental model

This is the single most important concept in Git. Until this clicks, commands feel random. Once it clicks, everything makes perfect sense.

Changes queued for the next commit
git commit
git commit
🗄
Repository
Permanent history stored in .git/
git log
example files in each state:
new-file.txt (untracked) style.css (staged) index.html (committed)
untracked / modified staged (index) committed (in history)
why does the staging area exist?

The staging area lets you craft precise, meaningful commits. You might change 5 files, but only want to commit 2 of them together. You stage just those 2, commit with a clear message, then stage the rest separately.

real-world example — selective staging
# You changed 3 files but they belong to different fixes $ git status Changes not staged for commit: modified: auth.js ← bug fix modified: style.css ← UI improvement modified: README.md ← documentation # Stage ONLY the bug fix first $ git add auth.js $ git commit -m "fix: resolve null pointer in auth check" # Then stage the rest separately $ git add style.css README.md $ git commit -m "docs: update README + improve button styles"
moving files between states
CommandWhat it does
git add <file>Working Dir → Staging: queue this file for next commit
git add .Stage ALL changed files in current directory
git add -pInteractively stage individual chunks (hunks) of changes
git commit -m "msg"Staging → Repo: permanently save all staged changes
git restore --staged <file>Unstage a file (move back from staging to working dir)
git restore <file>Discard working dir changes (dangerous — can't undo!)
git diffSee unstaged changes (working dir vs staging)
git diff --stagedSee staged changes (staging vs last commit)
🚫
Careful with git restore <file> — this throws away your unsaved working directory changes permanently. There is no undo. Always run git status first to see what will be lost.

Lesson 05 of 06 · Phase 1 — Foundation

add · commit · status · log · diff

These 5 commands are your daily bread. You'll run them dozens of times every day. Master them and you've mastered the core Git loop.

~45 min most-used commands
git status — your compass

git status tells you exactly where every file is. Run it constantly — before and after every other command. Think of it as asking "what's going on right now?"

git status output — reading it
$ git status On branch main Changes to be committed: ← STAGED (green in terminal) new file: index.html modified: style.css Changes not staged for commit: ← MODIFIED but not staged (red) modified: app.js Untracked files: ← Git doesn't know about these yet notes.txt temp/
git add — staging your changes
all ways to use git add
# Stage a single file $ git add index.html # Stage multiple files at once $ git add index.html style.css app.js # Stage everything in current directory (use with care) $ git add . # Stage all .js files $ git add *.js # Interactively choose which changes to stage (advanced, very useful) $ git add -p
git commit — saving a snapshot
writing good commits
# Commit with inline message (fast, most common) $ git commit -m "add navigation bar to homepage" # Stage + commit all tracked files in one step $ git commit -am "fix typo in login form" # Amend the LAST commit (fix message or add forgotten files) $ git commit --amend -m "correct commit message" # Successful commit output looks like this: [main 4f2a1b3] add navigation bar to homepage 2 files changed, 47 insertions(+), 3 deletions(-)
💡
Good commit message formula: Start with a verb in imperative mood. Keep it under 72 chars. Examples: "add user login form", "fix null pointer in auth", "remove deprecated API call". Avoid "fixed stuff" or "changes".
git log — reading history
git log — all the ways to read history
# Default: full log with dates, authors, full hashes $ git log # Compact: one commit per line (most useful daily) $ git log --oneline 4f2a1b3 (HEAD -> main) add navigation bar d8c3f91 fix typo in login form a1b2c3d initial commit # Visual branch graph (essential for Phase 2) $ git log --oneline --graph --all # Show commits by a specific author $ git log --author="Your Name" # Show last 5 commits only $ git log -5

git diff — seeing exact changes
reading a diff
$ git diff diff --git a/index.html b/index.html --- a/index.html +++ b/index.html @@ -10,7 +10,9 @@ <body> <h1>Hello World</h1> - <p>Old paragraph</p> ← removed (red) + <p>Updated paragraph</p> ← added (green) + <p>New second paragraph</p> </body>
exercise — the full git loop
Make 3 commits in your my-journal repo medium
  • 1In your my-journal repo: git add README.md → git commit -m "initial commit"
  • 2Create a new file: echo "Today I learned about git init" > day-1.md
  • 3Stage and commit: git add day-1.md → git commit -m "add day 1 journal entry"
  • 4Edit day-1.md, add another line. Then run git diff to see your change before staging.
  • 5Commit the update: git add -p (try interactive staging), then git commit -m "update day 1 notes"
  • 6Run git log --oneline — you should see all 3 commits.
Practice selective staging with 3 changed files medium
  • 1Create 3 new files: file-a.txt, file-b.txt, file-c.txt with any content
  • 2Run git status — all 3 should show as untracked
  • 3Stage only file-a and file-b: git add file-a.txt file-b.txt
  • 4Run git status again. See how file-a/b are green (staged) but file-c is still red (unstaged).
  • 5Commit: git commit -m "add file a and b"
  • 6Now stage and commit file-c separately with a different message.
  • 7Run git log --oneline — confirm you have 2 separate commits.
Lesson 06 of 06 · Phase 1 — Foundation

.gitignore — telling Git what to ignore

Not everything in your project should be tracked. Secrets, build artifacts, OS junk, and dependencies should stay out of Git. .gitignore is how you tell Git to look away.

~25 min hands-on best practice
what NOT to commit
🚫 Never commit these
.env / .env.local
API keys & passwords
node_modules/
__pycache__/
.DS_Store (macOS)
*.log files
dist/ / build/
IDE folders (.vscode/, .idea/)
✓ Always commit these
Source code (.js, .py, etc.)
package.json (not lock, debatable)
README.md
Config files (.eslintrc)
.gitignore itself
Tests
.env.example (template only)
Documentation
gitignore syntax — all the patterns
.gitignore
# Lines starting with # are comments # Ignore a specific file .env secrets.txt # Ignore all files with this extension *.log *.tmp *.pyc # Ignore an entire directory node_modules/ dist/ __pycache__/ .venv/ # Ignore files only in root, not subdirectories /config.json # Ignore all .txt files in a specific folder docs/*.txt # Use ! to UN-ignore something previously ignored *.log !important.log ← keep this one despite *.log above # Double star matches any depth **/temp ← matches temp/ anywhere in project

💡
gitignore.io — visit gitignore.io for instantly generated .gitignore templates for any stack. Also: gh repo create on GitHub lets you choose a template at repo creation time.
already tracked a file by mistake?
un-tracking a committed file
# If you accidentally committed .env and want to remove it from Git # (but keep the file on your disk) $ git rm --cached .env rm '.env' # Then add .env to .gitignore $ echo ".env" >> .gitignore # Commit both changes $ git add .gitignore $ git commit -m "stop tracking .env, add to gitignore"
🚫
If you pushed a secret to GitHub — act fast! The secret is now in public history. Immediately revoke/rotate the key in your service (AWS, Stripe, etc.), then remove it from Git history. History rewriting is Phase 5 territory — for now, just rotate the key immediately.
exercise — .gitignore in practice
Add a proper .gitignore to my-journal easy
  • 1In my-journal, create a file named .env and add API_KEY=supersecret123 inside it
  • 2Run git status — you'll see .env listed as untracked (don't add it!)
  • 3Create a .gitignore file. Use the builder above or type: .env, *.log, .DS_Store on separate lines
  • 4Run git status again — .env should now be invisible to Git. Only .gitignore shows as untracked.
  • 5Commit: git add .gitignore && git commit -m "add gitignore"
Phase 1 capstone — 10-commit journal project hard
  • 1Over the next 5 days, write one markdown file per day (day-1.md through day-5.md) in your my-journal repo
  • 2Each day: make at least 2 commits — one when you write, one when you edit
  • 3At least once, use git add -p to practice staging individual hunks
  • 4At least once, use git commit --amend to fix a typo in a commit message
  • 5At the end: run git log --oneline and verify you have 10+ commits with clear, descriptive messages
Goal of this exercise: You should feel totally comfortable with the add → commit → log loop. By day 5, these commands should feel like typing words — no thinking required.
