Lesson 01 of 06 · Phase 5 — Expert & Open Source

Git internals — the DAG

Most developers use Git for years without understanding what's underneath. Once you see the object model, every Git command becomes obvious — you'll never be confused by a weird state again.

~35 minexpert levelmind-bending
git's four object types

Everything in Git — files, directories, commits, tags — is stored as one of four object types in .git/objects/. Each object is identified by a SHA-1 hash of its contents.

📄
blob
File contents (no name, no path)
📁
tree
Directory listing (names + blob/tree refs)
🔵
commit
Points to a tree + parent(s) + metadata
🏷
tag
Annotated pointer to a specific commit
inspecting git objects — plumbing commands
git cat-file — peek inside any object
# Show the type of any object $ git cat-file -t HEAD commit # Show the content of a commit object $ git cat-file -p HEAD tree 8d3c5a1f9e2b47dc6a18f3e7d9c4b2a0e1f8d3c5 parent 4f2a1b3d8e9c7f6a5b4d3c2e1f0a9b8c7d6e5f4 author Your Name <you@email.com> 1700000000 +0000 committer Your Name <you@email.com> 1700000000 +0000 add user authentication module # Show what's in a tree (directory snapshot) $ git cat-file -p 8d3c5a1 100644 blob a8f3b2c1... .gitignore 100644 blob d4e5f6a7... README.md 040000 tree 9c8d7e6f... src 100644 blob 1a2b3c4d... package.json # Show a blob (file contents at a commit) $ git cat-file -p a8f3b2c1 node_modules/ .env dist/ # Show the full object graph for HEAD $ git log --oneline --graph --all
💡
Key insight: Git never stores diffs — it stores complete snapshots. A commit object points to a tree, which points to blobs (file contents). The magic: if a file hasn't changed, the tree just reuses the same blob hash. No duplication.
how branches and HEAD really work
.git/ — reading the raw files
# HEAD is just a text file pointing to a branch $ cat .git/HEAD ref: refs/heads/main # A branch is just a file containing a commit hash $ cat .git/refs/heads/main 4f2a1b3d8e9c7f6a5b4d3c2e1f0a9b8c7d6e5f4 # So "HEAD" = file → branch name → file → commit hash # That's the entire branch system. Just text files! # Detached HEAD = HEAD points directly to a commit, not a branch $ git checkout 4f2a1b3 # detach HEAD $ cat .git/HEAD 4f2a1b3d8e9c7f6a5b4d3c2e1f0a9b8c7d6e5f4 ← directly, no branch # Fix detached HEAD: create a branch where you are $ git switch -c my-recovery-branch

Lesson 02 of 06 · Phase 5 — Expert & Open Source

reflog · bisect · cherry-pick

The expert toolkit. reflog lets you undo anything. bisect finds bugs in O(log n) steps. cherry-pick copies specific commits anywhere. These three commands will save you countless hours.

~45 minpower commands
git reflog — your time machine & safety net

git reflog records every movement of HEAD — every checkout, commit, reset, merge. Even "deleted" commits live in reflog for 90 days. You can undo almost anything with it.

git reflog — recovering lost work
# Show all recent HEAD movements $ git reflog 4f2a1b3 HEAD@{0}: commit: add auth module d8c3f91 HEAD@{1}: checkout: moving from feature to main a1b2c3d HEAD@{2}: commit: WIP — login form 9c3e001 HEAD@{3}: commit (amend): fix typo in README c0ffee0 HEAD@{4}: reset: moving to HEAD~2 ← you reset here! # Recover commits after an accidental reset --hard $ git reset --hard a1b2c3d # go back to before the reset # OR create a branch at any reflog point $ git switch -c recovered HEAD@{3} # Show reflog for a specific branch $ git reflog show feature/login # Expire old reflog entries (careful!) $ git reflog expire --expire=30.days
✓
The golden rule: Whenever you think you've lost commits — run git reflog first. In 10 years of Git use, reflog has saved entire days of work. It's the real undo button.
git bisect — binary search for bugs

git bisect uses binary search to find the exact commit that introduced a bug. With 1000 commits of history, it finds the culprit in only 10 steps (log₂ 1000 ≈ 10).

git bisect — finding a bug in 10 steps
# Start bisect session $ git bisect start # Tell Git: current commit is BAD (bug is present) $ git bisect bad # or: git bisect bad HEAD # Tell Git: this old commit was GOOD (bug didn't exist) $ git bisect good v1.0.0 # or any known-good commit hash # Git checks out a commit in the middle. Test it! Bisecting: 512 revisions left to test (roughly 9 steps) # Run your test / reproduce the bug manually $ git bisect good # bug not here → go later $ git bisect bad # bug here → go earlier # After ~9 rounds, Git announces the culprit: a8f3b2c1 is the first bad commit Author: SomeContributor <dev@example.com> remove null check from user.getName() # Always finish the session $ git bisect reset # AUTOMATED bisect with a test script $ git bisect run npm test # auto: pass=good, fail=bad
git cherry-pick — transplant specific commits
git cherry-pick — pick individual commits
# Apply a single commit to your current branch $ git cherry-pick a8f3b2c1 # Apply a range of commits (a exclusive, b inclusive) $ git cherry-pick a8f3b2c..f7d9e3a # Pick without committing — inspect first $ git cherry-pick --no-commit a8f3b2c1 # Apply the fix from a hotfix branch to main $ git switch main $ git cherry-pick hotfix/critical-bug # picks branch tip commit # Real scenario: backport a feature to an older release $ git switch release/v1.x $ git cherry-pick a8f3b2c1 # transplant the fix from main # If conflict: resolve, then $ git cherry-pick --continue $ git cherry-pick --abort # or bail out entirely
⚠
Cherry-pick creates new commits with new hashes — the original commit still exists on its original branch. This is intentional (two branches, two records), but be aware when collaborating.

Lesson 03 of 06 · Phase 5 — Expert & Open Source

Submodules & subtrees

When your project depends on another Git repo — a shared library, a theme, a vendor SDK — submodules and subtrees let you include it with proper version pinning.

~30 minadvanced
git submodules — nested repos
submodule workflow
# Add a submodule $ git submodule add git@github.com:org/shared-lib.git libs/shared # Clone a repo that has submodules $ git clone --recurse-submodules git@github.com:you/project.git # Or initialize submodules in an already-cloned repo: $ git submodule update --init --recursive # Update all submodules to their latest commits $ git submodule update --remote --merge # See submodule status $ git submodule status -a8f3b2c libs/shared (v2.1.0) ← - means not initialized a8f3b2c libs/shared (v2.1.0) ← space means up to date +a8f3b2c libs/shared (v2.1.0) ← + means newer commit exists
⚠
Submodule gotcha: Submodules pin to a specific commit — not a branch. When the library updates, you must explicitly run git submodule update --remote and commit the pointer change. Forgetting this is the #1 submodule pain point.
git subtree — simpler alternative
git subtree — merge another repo as a directory
# Add another repo as a subdirectory $ git subtree add --prefix=libs/shared \ git@github.com:org/shared-lib.git main --squash # Pull upstream changes into the subtree $ git subtree pull --prefix=libs/shared \ git@github.com:org/shared-lib.git main --squash # Push changes back upstream (contribute back) $ git subtree push --prefix=libs/shared \ git@github.com:org/shared-lib.git main
💡
Submodule vs Subtree: Subtrees copy the code directly into your repo. No extra git submodule update step for teammates — they clone normally and everything just works. Use subtrees for simpler setups.
Lesson 04 of 06 · Phase 5 — Expert & Open Source

Git hooks

Hooks run scripts automatically at specific Git events — before a commit, before a push, after a merge. They enforce quality at the source: no bad code ever gets committed.

~35 minautomation
how hooks work

Hooks are executable scripts in .git/hooks/. Git runs them at specific lifecycle events. If a hook exits with a non-zero code, the operation is aborted. This is how you prevent bad commits before they happen.

.git/hooks/ — creating your first hook
# See all sample hooks $ ls .git/hooks/ pre-commit.sample commit-msg.sample pre-push.sample ... # Create a pre-commit hook $ cat > .git/hooks/pre-commit <<'EOF' #!/bin/sh echo "Running linter before commit..." npm run lint if [ $? -ne 0 ]; then echo "❌ Lint failed. Fix errors before committing." exit 1 # ← non-zero = abort the commit fi echo "✓ Lint passed" EOF # Make it executable (required!) $ chmod +x .git/hooks/pre-commit # Test it (runs when you git commit) $ git commit -m "test" Running linter before commit... ✓ Lint passed [main 4f2a1b3] test # Skip hooks when needed (emergency use only) $ git commit --no-verify -m "emergency fix"
sharing hooks — husky (the modern way)

The problem with .git/hooks/ is it's not committed to the repo — every teammate must manually install hooks. Husky solves this by storing hooks in .husky/ (which is committed) and auto-installing them on npm install.

husky + lint-staged — professional setup
# Install husky and lint-staged $ npm install --save-dev husky lint-staged $ npx husky init # pre-commit hook: only lint staged files $ echo "npx lint-staged" > .husky/pre-commit # commit-msg hook: enforce Conventional Commits $ echo 'npx commitlint --edit $1' > .husky/commit-msg # In package.json — lint-staged config { "lint-staged": { "*.{js,ts}": ["eslint --fix", "prettier --write"], "*.{css,md}": ["prettier --write"] } } # Now: every git commit auto-lints only changed files # Every teammate gets hooks automatically via npm install
Lesson 05 of 06 · Phase 5 — Expert & Open Source

Conventional Commits

A structured commit message format that enables automatic changelogs, version bumping, and semantic releases. Used by Angular, Vue, Electron, and thousands of open source projects.

~25 minindustry standard
the format
conventional commits specification
<type>[optional scope][!]: <description> [optional body] [optional footer(s)] ───────────────────────────────────────────────────── # Simple feature feat: add dark mode toggle to settings page # Bug fix with scope fix(auth): resolve null pointer when session expires # Breaking change (major version bump) feat(api)!: rename /user endpoint to /account BREAKING CHANGE: The /user endpoint has been renamed to /account. Update all API clients before upgrading. # Multi-line body fix(payment): prevent double-charge on timeout When a payment timed out, the webhook retry was not idempotent, causing a second charge. Added idempotency key to Stripe API call to prevent this. Closes #247

semantic-release GitHub Actions setup

Your first open source PR

The capstone. Everything you've learned — branching, PRs, commits, rebasing — comes together in a real open source contribution. This is how careers are built and communities are formed.

~60 mincapstonecareer milestone
where to find your first contribution
🔍 finding issues

  • → GitHub filter: label:"good first issue"
  • → up-for-grabs.net — community-maintained list
  • → Projects you already use — you know the pain points
  • → firstcontributions.github.io — practice repo
✓ types of first contributions
  • → Fix a typo in docs (no code needed)
  • → Improve error messages
  • → Add a missing test case
  • → Fix a small bug labeled "good first issue"
  • → Translate documentation
  • → Add code examples to docs
the open source etiquette rules
before you submit — the checklist
Before opening a PR to any open source project: ✓ Read the CONTRIBUTING.md file (every project has one) ✓ Read the CODE_OF_CONDUCT.md — understand community rules ✓ Check if there's already a PR for this issue ✓ Comment on the issue first: "I'd like to work on this" ✓ Fork the repo, don't work directly on the upstream ✓ Use a descriptive branch name (fix/typo-in-readme) ✓ Follow the project's coding style exactly ✓ Write tests if the project requires them ✓ Use Conventional Commits if the project uses them ✓ Keep the PR focused — one fix per PR ✓ Rebase onto upstream/main before submitting ✓ Write a clear PR description (what, why, how to test) ✓ Be patient — maintainers are volunteers. Wait ≥ 1 week. ✓ Respond politely to review feedback — don't argue style
exercise — submit your first real PR
Submit a PR to a real public repository capstone
  • 1Go to github.com/firstcontributions/first-contributions — a repo built for first PRs
  • 2Fork it. Clone your fork. Create branch add-YOUR-NAME
  • 3Follow their CONTRIBUTING.md to add your name to Contributors.md
  • 4Use a Conventional Commit: docs: add YOUR-NAME to contributors list
  • 5Push and open a PR. Write a proper description.
  • 6Once that's merged, find a real project you use. Search label:"good first issue"
  • 7Comment on an issue. Fork. Fix. PR. Wait. Celebrate.
